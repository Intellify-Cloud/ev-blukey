# Jekyll Static Site

Reusable static site setup built with Jekyll, Liquid, YAML data files, Sass, and a small Webpack asset pipeline. Use this structure for brochure sites, service sites, landing pages, and other mostly static projects.

## Setup

Install Ruby and Node dependencies:

```powershell
bundle install
npm i
```

## Commands

Build the Webpack bundle and Jekyll site:

```powershell
npm run build
```

Run the standard validation check:

```powershell
npm run check
```

Build front-end assets only:

```powershell
npm run bundle
```

Serve locally:

```powershell
npm run serve
```

Build assets, then serve locally:

```powershell
npm run dev
```

Serve with Jekyll trace output:

```powershell
npm run trace
```

## Project Shape

- Root `*.md` files define static pages.
- `_layouts/` contains page wrappers.
- `_includes/` contains reusable Liquid sections.
- `_data/` contains YAML content used by includes.
- `_assets/` contains source JavaScript, Sass, and Webpack-processed assets.
- `assets/` contains browser-facing static assets and compiled Webpack output.
- `_site/` is generated by Jekyll and should not be edited.

## Site Configuration

Set site-specific identity and SEO defaults in `_config.yml`, including values such as:

- `title`
- `description`
- `email`
- `phone`
- `author`
- `company`
- `url`
- `production_url`
- `logo`

Keep reusable section content in `_data/` where possible, then render it through Liquid includes.

## SEO Requirements

Every public page should have page-specific front matter:

```yaml
title: Page Title
description: "Clear search description for this page."
permalink: /page-url/
robots: index, follow
```

Global SEO defaults belong in `_config.yml`. Keep `url`, `production_url`, `title`, `description`, `company`, and `logo.path` accurate for the deployed site.

The shared head include should remain responsible for canonical URLs, Open Graph tags, Twitter card tags, favicon links, analytics loading, and JSON-LD schema. Do not duplicate that metadata inside individual pages.

For migrated pages, check that each page has one clear `h1`, useful alt text for meaningful images, crawlable links, and a concise description that matches the page content.

## Deployment Notes

This setup can deploy from committed static files or from a build step in CI. If the host builds the site, run `npm run build` during deployment. If the host serves the repository directly, make sure the compiled `assets/bundle.*` files and required static assets are committed intentionally.

Edit source files, then run `npm run build` before deployment.
